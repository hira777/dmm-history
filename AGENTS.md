# AGENTS.md

## 目的

このファイルは、このリポジトリで作業する人やエージェント向けの作業ルールをまとめたものです。
特に、Chrome 拡張の移行作業を安全に進めることを目的とします。

## このプロジェクトの前提

- Chrome 拡張のプロジェクトです
- 現状は古い構成です
  - Manifest V2
  - Vue 2
  - Webpack 4
  - TypeScript 3 系
- 今後は次の形を目指します
  - Manifest V3
  - React
  - TypeScript
  - 新しいビルド構成

## 最優先事項

1. まず Manifest V3 へ移行する
2. その後にビルド基盤を更新する
3. その後に Vue を React へ置き換える
4. 最後に依存関係を整理して上げ切る

依存関係の一括更新を最初にやらないこと。
理由は、問題が出た時に原因の切り分けが難しくなるためです。

## 作業前にやること

- 変更前に、何をやるかを短く説明する
- 影響するファイルを確認する
- 既存の変更を勝手に戻さない
- 大きな変更は一度にまとめて入れない

## 作業の進め方

### 1. 現状確認

着手前に、可能なら次を確認すること。

- `npm run build`
- `npm run lint`
- `npm run check-types`
- `npm test`

また、次の動作を手動で確認できるなら確認すること。

- 商品ページを開くと履歴が保存される
- ポップアップから履歴画面を開ける
- 履歴一覧が表示される
- 検索が動く
- 削除が動く

### 2. 実装順

次の順番を基本とすること。

1. MV3 対応
2. ビルド構成の更新
3. React の土台作成
4. Vue コンポーネントの置き換え
5. TypeScript の整理
6. テスト追加
7. 依存関係の整理
8. README 更新

### 3. 変更単位

1つのコミットや変更セットに、複数の大きな目的を混ぜないこと。

よい例:

- MV3 対応だけを行う
- popup の実装改善だけを行う
- Vue コンポーネント 1 画面分だけを React 化する

避けたい例:

- MV3 対応と React 化と依存更新を同時に行う

## コード変更時の注意

- 既存のロジックを変える時は、挙動差分を明確にする
- DOM 取得とデータ整形の責務をできるだけ分ける
- 純粋関数にできる処理は切り出す
- 強い型変換をなるべく減らす
- Chrome API の利用箇所は分かりやすく保つ

## テスト方針

- まずは純粋関数からテストを追加する
- 次を優先して守る
  - 履歴追加
  - 重複排除
  - 最大件数の制御
  - 検索条件の判定
- UI テストは後から最小限でよい
- `describe` と `it` の文言は日本語にする

## React 化の方針

- React では関数コンポーネントを使う
- `.vue` ではなく `.tsx` へ寄せる
- 状態管理はまず React 標準の state で足りるかを確認する
- 重い状態管理ライブラリは必要になるまで入れない

## 依存関係の方針

- 依存更新は段階的に行う
- 不要な依存は放置せず削除する
- `node-sass` は継続利用しない
- Vue 系依存は React 化完了後に削除する

## ブランチ運用

この移行作業の推奨ブランチ名:

- `chore/react-ts-mv3-migration`

小さい単位で枝を分ける場合の例:

- `chore/mv3-migration`
- `chore/build-refresh`
- `feat/react-history-page`

## 最後の確認

変更後は、可能な範囲で次を確認すること。

- ビルドが通る
- 型チェックが通る
- lint が通る
- テストが通る
- 拡張機能として最低限の手動確認が終わっている

確認できていないものがあれば、作業報告で明記すること。
